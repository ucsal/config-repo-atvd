# config-repo-atvd
O projeto Spring Cloud Config tem como propósito fundamental oferecer um serviço de configuração centralizado e versionado para aplicações distribuídas, funcionando como uma fonte única de verdade para todos os parâmetros de runtime dos microsserviços. Sua principal funcionalidade é carregar arquivos de configuração (geralmente .yml ou .properties) de um repositório Git e entregá-los aos Config Clients via HTTP, garantindo que o comportamento de um microsserviço seja determinado pelo ambiente em que está rodando, e não pelo seu código hardcoded.

Sua necessidade no cenário de um banco digital é crítica e se justifica pela separação de preocupações e pela segurança. Em um ambiente com múltiplos serviços e diferentes estágios (desenvolvimento, homologação, produção), o Config Server garante a consistência ao mapear o serviço certo com as configurações certas. Isso evita erros manuais e garante que credenciais de produção (como URLs de banco de dados e chaves de API) sejam usadas apenas no ambiente de produção. Além disso, ao utilizar o Git, o sistema bancário ganha rastreabilidade e capacidade de auditoria de todas as alterações de configuração, o que é vital para a conformidade regulatória.


a) Explique o conceito de configuração externalizada e centralizada.
A configuração externalizada é o princípio de design de software que dita que os parâmetros operacionais de uma aplicação (como portas, URLs de serviços externos, e credenciais) devem ser armazenados fora do código-fonte e do artefato de build final, como um arquivo .jar. Isso é necessário para aderir ao princípio do The Twelve-Factor App, permitindo que o mesmo artefato binário possa ser promovido por diferentes ambientes (dev, homologação, prod) sem ser reempacotado, alterando seu comportamento apenas pela fonte de configuração que consome. Já a configuração centralizada leva este conceito adiante, consolidando todos os arquivos de configuração de todos os microsserviços em um único local gerenciado (o Config Server, que usa o Git como backend). Essa centralização simplifica a governança, a aplicação de políticas de segurança e o gerenciamento de configurações que são compartilhadas por vários serviços dentro do ecossistema bancário.


b) Por que é importante ter um Config Server em um sistema bancário com múltiplos ambientes (dev, homologação, produção)?
É de extrema importância ter um Config Server em um sistema bancário devido à necessidade rigorosa de segregação de dados e recursos entre os ambientes. O Config Server, utilizando os perfis de ambiente (dev, homologação, prod), garante que o account-service em desenvolvimento utilize um banco de dados de teste (com dados dummy e senhas de acesso de baixo privilégio), enquanto o mesmo serviço em produção utilize a URL e as credenciais do banco de dados principal. Isso é crucial para a segurança e conformidade. Além disso, a centralização no Config Server atua como um ponto de controle que pode integrar-se ao fluxo de CI/CD (Integração e Entrega Contínua), garantindo que apenas configurações validadas cheguem ao ambiente de produção, minimizando o risco de falhas operacionais devido a configurações incorretas.
